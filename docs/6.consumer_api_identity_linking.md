## Overview

### Purpose

This document defines how **Consumer App users** are linked to:
1. **Supplier System** (for Market ordering - B2C and B2B)
2. **POS V2 System** (for restaurant ownership/management verification - B2B only)

### Critical Principle: TWO SEPARATE FLOWS

**There are TWO distinct Market ordering flows:**

#### **Flow A: Personal Account (B2C - Retail)**
```
Consumer User → Supplier API (Direct Link)
     ↓                    ↓
  Personal         Personal Supplier
  Account          Customer Account
```

**Use Case**: User buying groceries for their home
- **NO POS V2 verification required**
- Direct link using personal phone number
- Retail pricing
- Immediate payment required

---

#### **Flow B: Merchant Account (B2B - Wholesale)**
```
Consumer User → POS V2 Restaurant Verification → Supplier Account Linking
     ↓                    ↓                              ↓
  Personal         Proves ownership              Restaurant's
  Account          of Restaurant                 Supplier Account
```

**Use Case**: Restaurant owner buying supplies for business
- **POS V2 verification required first**
- Link using restaurant's phone number
- Wholesale pricing
- Credit terms available

---

**Why this distinction matters:**
- Personal users can order immediately without restaurant verification
- Merchant users get wholesale pricing only after proving restaurant ownership
- Each restaurant has its own Supplier customer account
- Wholesale pricing and credit terms are per-restaurant
- Multiple users can manage the same restaurant's Supplier account

## Linking Flow Sequence

### Complete Linking Journeys

#### **Journey A: Personal User (B2C - Retail Orders)**

```
Step 1: User Downloads Consumer App
    ↓
Step 2: OTP Login (Personal Account Created)
    ↓
Step 3: User Opens Market Tab (First Time)
    ↓
Step 4: Supplier Account Auto-Created (Using Personal Phone)
    ↓
Step 5: User Can Order at Retail Prices Immediately
```

**Timeline**: 2-3 seconds (automatic, transparent to user)

---

#### **Journey B: Merchant User (B2B - Wholesale Orders)**

```
Step 1: User Downloads Consumer App
    ↓
Step 2: OTP Login (Personal Account Created)
    ↓
Step 3: User Wants Wholesale Access in Market
    ↓
Step 4: User Links Restaurant (POS V2 Verification)
    ↓
Step 5: Supplier Account Automatically Linked (Using Restaurant Phone)
    ↓
Step 6: User Can Order at Wholesale Prices
```

**Timeline**: 2-3 minutes (requires manual code entry)

**Trigger**: Personal user opens Market tab for first time

```
    ↓
   Consumer API → Supplier API
    ↓
```

- User has personal Supplier customer account

**Key Points**:
- ✅ **Automatic**: Happens transparently during first checkout
- ✅ **Instant**: No manual verification required
- ✅ **Separate Identity**: Different from restaurant Supplier accounts
- ✅ **Retail Pricing**: No wholesale discounts
- ✅ **Immediate Payment**: No credit terms

**Trigger**: User wants **wholesale pricing**, clicks "Link Restaurant" in app

**Use Case**: Restaurant owner wants to buy supplies at wholesale prices with credit terms

**Important**: This flow is ONLY for B2B (wholesale). Personal users can skip this and order at retail prices using Flow 0.

```
    ↓
POS V2 generates unique code:
    ↓
```

#### Step 2B: User Links Restaurant (Consumer App Side)

```
Consumer App
    ↓
User navigates to: Profile → Link Restaurant
    ↓
User enters code: LINK-BBQ456-7D89F2
    ↓
Mobile App → Consumer API
    ↓
POST /v1/users/merchant/link

{
  "verificationCode": "LINK-BBQ456-7D89F2"
}
    ↓
Consumer API Logic:
1. Extract code: "LINK-BBQ456-7D89F2"
2. Call POS V2 API to verify code
    ↓
    Consumer API → POS V2 API
    ↓
    POST /api/restaurant-link-codes/verify
    {
      "code": "LINK-BBQ456-7D89F2",
      "consumerUserId": "consumer_123",
      "consumerPhone": "8562093352677"
    }
    ↓
    POS V2 Logic:
    - Find code in database
    - Check: code exists? ✓
    - Check: not expired? ✓
    - Check: not used? ✓
    - Check: user's phone matches POS staff? (Optional security)
    ↓
    POS V2 Response:
    {
      "valid": true,
      "restaurant": {
        "id": "rest_456",
        "name": "Noy's BBQ Shop",
        "address": "123 Main St, Vientiane"
      },
      "role": "owner",
      "permissions": ["order", "view_reports", "manage_staff"]
    }
    ↓
3. Mark code as used in POS V2:
   {
     code: "LINK-BBQ456-7D89F2",
     used: true,
     usedBy: "consumer_123",
     usedAt: Date.now()
   }
    ↓
4. Update Consumer user:
   user.merchantProfiles.push({
     restaurantId: "rest_456",
     restaurantName: "Noy's BBQ Shop",
     role: "owner",
     verifiedAt: new Date(),
     supplierCustomerId: null,  ← Will be set in next step
     permissions: ["order", "view_reports", "manage_staff"]
   })
   user.roles.push("merchant_owner")
    ↓
Consumer API Response:
{
  "message": "Restaurant linked successfully",
  "profiles": [
    {
      "type": "personal",
      "name": "Somsack (Personal)",
      "isActive": true
    },
    {
      "type": "merchant",
      "restaurantId": "rest_456",
      "restaurantName": "Noy's BBQ Shop",
      "role": "owner",
      "isActive": false,
      "supplierLinked": false  ← Not yet linked to Supplier
    }
  ]
}
```

**Result**: User is now verified as restaurant owner in POS V2, but Supplier account not linked yet.

### Flow 3: Restaurant Supplier Account Linking (Automatic) - B2B Only

**Trigger**: User switches to Merchant profile and accesses Market for first time

**Use Case**: After restaurant verification, link restaurant to its Supplier wholesale account

**Note**: This is different from Flow 0 (personal Supplier linking). This creates/links a restaurant's wholesale Supplier account.

```
Consumer App
    ↓
User taps: Profile Switcher → "Noy's BBQ Shop (Owner)"
    ↓
POST /v1/auth/switch-profile
{
  "profileType": "merchant",
  "restaurantId": "rest_456"
}
    ↓
Consumer API updates: user.activeProfile = "merchant"
    ↓
User navigates to: Market Tab
    ↓
App requests: GET /v1/market/products
    ↓
Consumer API Logic:
1. Check: user.activeProfile === "merchant"? ✓
2. Get active merchant profile:
   merchantProfile = user.merchantProfiles.find(p => p.restaurantId === "rest_456")
3. Check: merchantProfile.supplierCustomerId exists?
    ↓
   NO → Trigger Supplier Linking Flow
    ↓
   Consumer API calls POS V2: Get Restaurant Supplier Info
    ↓
   GET /api/restaurants/rest_456/supplier-info
    ↓
   POS V2 checks: Does this restaurant have Supplier account?
    ↓
   POS V2 Response (Case A - Restaurant has existing Supplier account):
   {
     "hasSupplierAccount": true,
     "supplierCustomerId": "supp_cust_789",
     "supplierCustomerPhone": "2093352677",
     "createdAt": "2024-01-15T10:00:00.000Z"
   }
    ↓
   Consumer API: Link to existing Supplier account
   merchantProfile.supplierCustomerId = "supp_cust_789"
   user.save()
    ↓
   OR
    ↓
   POS V2 Response (Case B - Restaurant has NO Supplier account):
   {
     "hasSupplierAccount": false,
     "restaurantPhone": "8562012345678",
     "restaurantName": "Noy's BBQ Shop"
   }
    ↓
   Consumer API: Create NEW Supplier account for restaurant
    ↓
   Consumer API → Supplier API
    ↓
   POST /user/authenticate/token
   {
     "phone": "2012345678",  ← Restaurant's phone (normalized)
     "exchangeKey": SUPPLIER_EXCHANGE_KEY
   }
    ↓
   Supplier API Logic:
   - Check if customer exists with this phone
   - If NO: Auto-create new customer
   - Return customer ID and token
    ↓
   Supplier API Response:
   {
     "accessToken": "...",
     "customerId": "supp_cust_new_123",
     "user": {
       "_id": "supp_cust_new_123",
       "phoneNumber": "2012345678",
       "role": "CUSTOMER"
     }
   }
    ↓
   Consumer API updates:
   merchantProfile.supplierCustomerId = "supp_cust_new_123"
   user.save()
    ↓
   Consumer API notifies POS V2: Store Supplier customer ID
    ↓
   POST /api/restaurants/rest_456/supplier-link
   {
     "supplierCustomerId": "supp_cust_new_123",
     "supplierPhone": "2012345678"
   }
    ↓
   POS V2 stores:
   {
     restaurantId: "rest_456",
     supplierCustomerId: "supp_cust_new_123",
     linkedAt: Date.now()
   }
    ↓
4. Now fetch products using restaurant's Supplier account:
   - Get token for supplierCustomerId
   - Call Supplier API with this token
   - Return wholesale products
```

**Result**: Restaurant is now linked to Supplier, user can order at wholesale prices.

#### 2. Verify Restaurant Link Code

**Endpoint**: `POST /api/restaurant-link-codes/verify`

**Purpose**: Validate code submitted by Consumer API

**Authentication**: API Key (from Consumer API)

**Request**:
```json
{
  "code": "LINK-BBQ456-7D89F2",
  "consumerUserId": "consumer_123",
  "consumerPhone": "8562093352677"
```

```json

  "valid": true,
```


- `CODE_NOT_FOUND`: Code doesn't exist

**Logic**:
1. Find code in database
2. Check not expired
3. Check not used
4. Mark as used
5. Return restaurant details

## Supplier API Linking

### Key Principle: TWO Types of Supplier Accounts

**Important**: Users can have BOTH a personal Supplier account AND restaurant Supplier accounts.

#### **Comparison: Personal vs Restaurant Supplier Accounts**

| Aspect | Personal Account (B2C) | Restaurant Account (B2B) |
|--------|----------------------|-------------------------|
| **Stored In** | `user.supplierId` | `user.merchantProfiles[].supplierCustomerId` |
| **Created When** | First Market checkout (personal profile) | After restaurant linking (merchant profile) |
| **Linked To** | User's personal phone | Restaurant's phone |
| **Pricing** | Retail (higher) | Wholesale (lower) |
| **Payment** | Immediate (Phapay) | Immediate or Credit Terms |
| **POS Verification** | ❌ Not required | ✅ Required |
| **Orders Under** | Personal name | Restaurant name |
| **Delivered To** | Personal address | Restaurant address |
| **Invoice To** | Personal (optional) | Business (with tax ID) |
| **Multiple Users** | ❌ No (one per person) | ✅ Yes (many users per restaurant) |

**Example - Same User, Two Supplier Accounts:**
```javascript
{
  _id: "consumer_123",
  phone: "8562093352677",
  fullName: "Somsack Phonevilay",
  
  // Personal Supplier account (for home shopping)
  supplierId: "supp_cust_456",
  
  // Restaurant Supplier accounts (for business)
  merchantProfiles: [
    {
      restaurantId: "rest_789",
      restaurantName: "Noy's BBQ Shop",
      supplierCustomerId: "supp_cust_999"  // Different from personal!
    }
  ]
}
```

**When User Orders:**
- **Personal Profile Active**: Orders using `user.supplierId` → Retail prices
- **Merchant Profile Active**: Orders using `merchantProfile.supplierCustomerId` → Wholesale prices

### Supplier Account Strategy

#### Option A: Use Restaurant Phone Number
```javascript
// Restaurant phone as Supplier customer
restaurantPhone = "8562012345678"
normalizedPhone = "2012345678"  // For Supplier API

// Authenticate Supplier API:
POST /user/authenticate/token
{
  "phone": "2012345678",
  "exchangeKey": SUPPLIER_EXCHANGE_KEY
}

// Result: Restaurant gets its own Supplier customer account
supplierCustomerId = "supp_cust_rest456"
```

**Pros**:
- Clear separation (restaurant orders vs personal orders)
- Easy to track restaurant purchase history
- Credit terms applied per restaurant

**Cons**:
- Each restaurant needs unique phone number

---

#### Option B: Use Derived Phone Number
```javascript
// If restaurant shares phone with owner:
ownerPhone = "8562093352677"
restaurantId = "rest_456"

// Create unique identifier:
derivedPhone = `${ownerPhone}-${restaurantId}`
// Example: "8562093352677-rest456"

// Or use restaurant's tax ID / business registration number
```

### Supplier Authentication Flow

#### **Authentication for Personal User (B2C)**

```javascript
// Consumer API Service - supplierApi.service.js

async authenticateForPersonalUser(userId) {
  const user = await User.findById(userId);
  
  // Check if user has personal Supplier account
  if (user.supplierId) {
    // Check cache
    const cacheKey = `supplier:token:personal:${userId}`;
    const cachedToken = await redis.get(cacheKey);
    if (cachedToken) return cachedToken;
  }
  
  // Normalize phone for Supplier API
  const normalizedPhone = normalizePhoneForSupplier(user.phone);
  
  // Authenticate (auto-creates customer if not exists)
  const response = await supplierClient.post('/user/authenticate/token', {
    phone: normalizedPhone,
    exchangeKey: SUPPLIER_EXCHANGE_KEY
  });
  
  // Store Supplier customer ID if new
  if (!user.supplierId) {
    user.supplierId = response.data.customerId;
    await user.save();
  }
  
  // Cache token (24h)
  await redis.setex(
    `supplier:token:personal:${userId}`,
    86400,
    response.data.accessToken
  );
  
  return response.data.accessToken;
}
```

```javascript
async authenticateForRestaurant(restaurantId) {
  // 1. Get restaurant details from POS V2
  const restaurant = await posV2Api.getRestaurant(restaurantId);
  
  // 2. Check if restaurant has existing Supplier account
  const supplierInfo = await posV2Api.getRestaurantSupplierInfo(restaurantId);
  
  if (supplierInfo.hasSupplierAccount) {
    // Use existing account
    const phone = supplierInfo.supplierCustomerPhone;
    
    // Check cache
    const cacheKey = `supplier:token:restaurant:${restaurantId}`;
    const cachedToken = await redis.get(cacheKey);
    if (cachedToken) return cachedToken;
    
    // Authenticate
    const response = await supplierClient.post('/user/authenticate/token', {
      phone: normalizePhoneForSupplier(phone),
      exchangeKey: SUPPLIER_EXCHANGE_KEY
    });
    
    // Cache token (24h)
    await redis.setex(cacheKey, 86400, response.data.accessToken);
    
    return response.data.accessToken;
    
  } else {
    // Create new Supplier account for restaurant
    const phone = restaurant.phone;
    const normalizedPhone = normalizePhoneForSupplier(phone);
    
    // Authenticate (auto-creates customer if not exists)
    const response = await supplierClient.post('/user/authenticate/token', {
      phone: normalizedPhone,
      exchangeKey: SUPPLIER_EXCHANGE_KEY
    });
    
    // Store Supplier customer ID in POS V2
    await posV2Api.linkSupplierCustomer(restaurantId, {
      supplierCustomerId: response.data.customerId,
      supplierPhone: normalizedPhone
    });
    
    // Cache token
    await redis.setex(
      `supplier:token:restaurant:${restaurantId}`,
      86400,
      response.data.accessToken
    );
    
    return response.data.accessToken;
  }
}
```

### Supplier Token Management

**Key Strategy**: Different caching for personal vs restaurant accounts

```javascript
// Redis Cache Keys:

// Personal account tokens (per user)
supplier:token:personal:{userId}  → Token for user's personal Supplier account
// Example:
supplier:token:personal:user_123 → "eyJhbGc..."

// Restaurant account tokens (per restaurant)
supplier:token:restaurant:{restaurantId}  → Token for restaurant's Supplier account
supplier:customerId:restaurant:{restaurantId}  → Supplier customer ID
// Example:
supplier:token:restaurant:rest_456 → "eyJhbGc..."
supplier:customerId:restaurant:rest_456 → "supp_cust_789"
```

**Why different strategies**:

**Personal Tokens (per user)**:
- Each user has own Supplier account
- Orders under personal identity
- Cannot be shared
- Retail pricing

**Restaurant Tokens (per restaurant)**:
- Multiple users share same restaurant account
- All users use same Supplier account
- Order history consolidated per restaurant
- Credit terms applied to restaurant, not individual
- Wholesale pricing

```javascript
// Endpoint: POST /v1/auth/switch-profile
async switchProfile(req, res) {
  const { profileType, restaurantId } = req.body;
  const userId = req.user.id;
  
  const user = await User.findById(userId);
  
  if (profileType === 'personal') {
    user.activeProfile = 'personal';
    user.activeRestaurantId = null;
    await user.save();
    
    return res.json({
      activeProfile: {
        type: 'personal',
        viewMode: 'retail'
      }
    });
  }
  
  if (profileType === 'merchant') {
    // Find restaurant in user's profiles
    const profile = user.merchantProfiles.find(
      p => p.restaurantId === restaurantId
    );
    
    if (!profile) {
      return res.status(403).json({
        error: 'You do not have access to this restaurant'
      });
    }
    
    user.activeProfile = 'merchant';
    user.activeRestaurantId = restaurantId;
    await user.save();
    
    return res.json({
      activeProfile: {
        type: 'merchant',
        restaurantId: profile.restaurantId,
        restaurantName: profile.restaurantName,
        role: profile.role,
        viewMode: 'wholesale',
        supplierLinked: !!profile.supplierCustomerId
      }
    });
  }
}
```

```javascript
// Middleware: validateMerchantAccess
async validateMerchantAccess(req, res, next) {
  const userId = req.user.id;
  const user = await User.findById(userId);
  
  // Check: User in merchant mode?
  if (user.activeProfile !== 'merchant') {
    return res.status(403).json({
      error: 'Wholesale access requires merchant profile'
    });
  }
  
  // Check: User has restaurant linked?
  const activeRestaurant = user.merchantProfiles.find(
    p => p.restaurantId === user.activeRestaurantId
  );
  
  if (!activeRestaurant) {
    return res.status(403).json({
      error: 'No active restaurant profile'
    });
  }
  
  // Check: Restaurant has Supplier account?
  if (!activeRestaurant.supplierCustomerId) {
    return res.status(400).json({
      error: 'Supplier account not linked',
      action: 'Please complete Supplier linking'
    });
  }
  
  // Check: User has permission to order?
  if (!activeRestaurant.permissions.includes('order')) {
    return res.status(403).json({
      error: 'You do not have permission to place orders'
    });
  }
  
  // Attach restaurant context to request
  req.restaurant = {
    id: activeRestaurant.restaurantId,
    name: activeRestaurant.restaurantName,
    supplierCustomerId: activeRestaurant.supplierCustomerId,
    role: activeRestaurant.role
  };
  
  next();
}
```

### Retry Logic

```javascript
// Background job: Retry failed Supplier linking
async retryFailedSupplierLinks() {
  const usersWithFailedLinks = await User.find({
    'merchantProfiles.supplierCustomerId': null,
    'merchantProfiles.supplierLinkRetries': { $lt: 3 }
  });
  
  for (const user of usersWithFailedLinks) {
    for (const profile of user.merchantProfiles) {
      if (!profile.supplierCustomerId) {
        try {
          // Retry linking
          const token = await supplierApi.authenticateForRestaurant(
            profile.restaurantId
          );
          
          // Success - update profile
          profile.supplierCustomerId = token.customerId;
          profile.supplierLinkRetries = 0;
          await user.save();
          
          logger.info('Supplier link retry succeeded', {
            userId: user._id,
            restaurantId: profile.restaurantId
          });
          
        } catch (error) {
          profile.supplierLinkRetries = (profile.supplierLinkRetries || 0) + 1;
          await user.save();
          
          logger.error('Supplier link retry failed', {
            userId: user._id,
            restaurantId: profile.restaurantId,
            retries: profile.supplierLinkRetries
          });
        }
      }
    }
  }
}

// Run every 30 minutes
cron.schedule('*/30 * * * *', retryFailedSupplierLinks);
```

## Summary

### Key Takeaways

1. **TWO Separate Flows**:
   - **B2C (Personal)**: Consumer API → Supplier API (direct, no POS verification)
   - **B2B (Merchant)**: Consumer API → POS V2 → Supplier API

2. **TWO Types of Supplier Accounts**:
   - **Personal Account**: `user.supplierId` (for home shopping, retail prices)
   - **Restaurant Accounts**: `merchantProfile.supplierCustomerId` (for business, wholesale prices)

3. **Personal Users**: Can order immediately without restaurant verification (retail pricing)

4. **Merchant Users**: Must prove restaurant ownership first (wholesale pricing + credit terms)

5. **Automatic Linking**: Both personal and restaurant Supplier accounts are linked automatically on first use

6. **Multi-Restaurant**: Users can manage multiple restaurants, each with separate Supplier accounts

7. **Same User, Multiple Accounts**: One user can have 1 personal account + N restaurant accounts
